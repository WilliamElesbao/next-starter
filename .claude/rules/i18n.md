# Internationalization (i18n)

## Stack

`next-intl` v4 with locale files for `en` (source of truth) and `pt-BR`.

## File Structure

```
src/lib/i18n/
├── global.ts              # Global i18n type declarations
├── navigation.ts          # i18n-aware Link, redirect, useRouter
├── requests.ts            # Server-side message loading
├── routing.ts             # Locale configuration
└── locales/
    ├── en.json            # English (source of truth)
    ├── en.d.json.ts       # Auto-generated TypeScript types
    └── pt-BR.json         # Portuguese (Brazil)
```

## Usage

### Client Component

```tsx
"use client"
import { useTranslations } from "next-intl"

function SignInForm() {
  const t = useTranslations("common")
  return <input placeholder={t("email")} />
}
```

### Server Component

```tsx
import { getTranslations } from "next-intl/server"

async function Page() {
  const t = await getTranslations("dashboard")
  return <h1>{t("title")}</h1>
}
```

### Navigation — Never use `next/link` or `next/navigation` directly

```tsx
import { Link, useRouter, redirect } from "@/lib/i18n/navigation"

// ✅
<Link href="/dashboard">Dashboard</Link>

// ❌
import Link from "next/link"
```

## Localized Validation (Zod v4)

```ts
// form-schema.ts
import { useTranslations } from "next-intl"
import { z } from "zod"

export const useSignInFormSchema = () => {
  const t = useTranslations("validation")

  return z.object({
    email: z.email({ error: t("email.please-enter-a-valid-email") }),
    password: z.string().min(1, { error: t("password.password-is-required") }),
  })
}
```

Zod v4 passes custom errors via `{ error: "..." }` key.

## Adding New Keys

```
1. Add key to en.json (source of truth)
2. Add translated key to pt-BR.json
3. Run `bun run locale-check` — must pass
4. TypeScript types update automatically
```

## Validation Scripts

```bash
bun run locale-check       # Key parity between en.json and pt-BR.json
bun run locale-unused      # Detect orphan keys in translation files
```

Both must pass in CI — no missing or orphan keys allowed.

## Key Naming Convention

- Dot notation: `dashboard.toast.welcome.title`
- Keys mirror the English sentence in kebab-case
- Punctuation dropped, proper-noun casing preserved
- Rename a key when the English copy changes meaningfully

## Translation Files

```json
{
  "common": {
    "email": "Email",
    "password": "Password",
    "login": "Login"
  },
  "validation": {
    "email": { "please-enter-a-valid-email": "Please enter a valid email address" },
    "password": { "password-is-required": "Password is required" }
  }
}
```

## Rules

| Rule | Enforcement |
|---|---|
| Every rendered string must be in translation files | Required |
| New keys added to `en.json` first, then `pt-BR.json` | Required |
| Navigation via `@/lib/i18n/navigation` | Required — never `next/link` directly |
| `useTranslations` in Client Components | Required |
| `getTranslations` in Server Components (await) | Required |
| Run `locale-check` after any key changes | Required |
